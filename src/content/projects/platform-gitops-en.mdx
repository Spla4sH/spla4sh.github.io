## What it's about

I wanted to understand ArgoCD's App-of-Apps pattern not just in theory but in practice — on a realistic setup that could plausibly run in production. The result is a local lab with two repositories: one for the application ([demo-app](https://github.com/Spla4sH/demo-app), Go + nginx) and one for the GitOps state ([platform-gitops](https://github.com/Spla4sH/platform-gitops)).

## Architecture

- **kind** cluster running locally in Docker Desktop
- **ArgoCD** running in-cluster, watching the `platform-gitops` repo
- **App-of-Apps**: a root Application references further Application manifests — adding a new service just means committing a YAML in the right folder
- **In-repo Helm chart** for the demo app, parameterised per environment
- **CI** in `demo-app` builds container images to `ghcr.io/spla4sh/demo-app-*` and triggers an image bump in the GitOps repo

## What I learned

The power of GitOps only shows when you apply it consistently: drift is immediately visible, rollbacks are a `git revert`. The discipline of _never_ making `kubectl` changes to the cluster is unfamiliar at first — but that's exactly the point.
