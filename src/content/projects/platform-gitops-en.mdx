## What it's about

I want to understand ArgoCD's App-of-Apps pattern not just in theory but in practice, on a realistic setup that could plausibly run in production. The lab consists of two repositories: [demo-app](https://github.com/Spla4sH/demo-app) for the application code (Go API + nginx frontend) and [platform-gitops](https://github.com/Spla4sH/platform-gitops) as the GitOps source of truth watched by ArgoCD.

## The design

The target architecture is documented in [ARCHITECTURE.md](https://github.com/Spla4sH/platform-gitops/blob/main/ARCHITECTURE.md):

- **kind** cluster running locally in Docker, with port mappings for 80/443
- **App-of-Apps**: a root Application scans the apps directory: infrastructure (ingress-nginx, cert-manager, sealed-secrets, monitoring) and workloads each get their own Application manifest
- **In-repo Helm chart** for the demo app
- **CI** in `demo-app` builds container images to GHCR and triggers the image bump in the GitOps repo

## Status & next steps

The architecture documentation and the kind cluster configuration are in place, the repos are set up. Now comes the actual implementation:

1. Bootstrap script (kind create → ArgoCD install → root app)
2. Application manifests for infra and workloads
3. Helm chart for the demo app
4. CI workflow with image build and automated image bump

## Why this project

GitOps promises: drift is immediately visible, rollbacks are a `git revert`, and nothing on the cluster gets changed by hand. Whether that discipline holds up day to day is something you only find out by building the whole setup yourself.
