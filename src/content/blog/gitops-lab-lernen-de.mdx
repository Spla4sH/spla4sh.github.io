Ich habe in den letzten Wochen ein lokales Kubernetes-Lab gebaut, um das **App-of-Apps-Pattern** von ArgoCD wirklich zu verstehen — nicht aus einem Tutorial, sondern in einem Setup, das so auch in Produktion stehen könnte. Hier sind die drei Sachen, die mich am meisten überrascht haben.

## 1. GitOps ist eine Disziplin, kein Tool

Man kann ArgoCD installieren und trotzdem kein GitOps machen. Der Moment, in dem ich das verstanden habe, war beim Debuggen: Ich war frustriert, dass ein Deployment nicht hochkam, hab kurz `kubectl edit deployment` gemacht — und dann ArgoCD das _zurückreverten_ sehen.

Das ist nicht ArgoCDs Bug. Das ist genau der Punkt. Wenn das Repo die Wahrheit ist, dann muss jede Änderung dort beginnen. Sonst ist Drift in Sekunden zurück.

## 2. App-of-Apps macht das Plattform-Repo zur Karte

Die Versuchung ist groß, alle Manifeste in einen großen Ordner zu werfen. App-of-Apps zwingt dich zu einer Hierarchie:

```
clusters/local/
├── root.yaml             # zeigt auf den Ordner darunter
└── applications/
    ├── demo-app.yaml
    └── ...
```

Jedes neue Service ist genau ein neuer YAML-File in `applications/`. Plötzlich ist das Repo lesbar.

## 3. Die Image-Update-Schleife ist die unsichtbare Komplexität

Code-Repo → CI → Container-Image → Image-Bump-Commit im GitOps-Repo → ArgoCD reconciliert.

Diese Kette klingt simpel, hat aber eine Menge Stolperfallen — Tag-Strategien, Race-Conditions zwischen Commits, Permissions zwischen Repos. Ich hatte sie initial mit einer GitHub Action gelöst, die einen Cross-Repo-Commit macht. Funktioniert, aber für ein Produktions-Setup würde ich heute Argo CD Image Updater nehmen.
